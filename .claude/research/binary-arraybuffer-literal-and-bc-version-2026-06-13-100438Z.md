# Binary ArrayBuffer literal + BC_VERSION / opcode-addition gotchas

Non-obvious findings from adding the non-standard binary ArrayBuffer literal syntax (`SOH(0x01) <decimal byte-length N> STX(0x02) <N raw bytes> ETX(0x03)`, evaluates to a fresh ArrayBuffer).

## Adding any opcode renumbers the SHORT opcodes, which DO serialize

`OP_TEMP_START = OP_nop + 1` (quickjs.c, in the `OPCodeEnum`), and the short opcodes (`push_minus1`, `get_loc0`, etc., under `#if SHORT_OPCODES` in quickjs-opcode.h) are numbered in the temp range via the `short_opcode_info()` macro. Short opcodes are emitted into final bytecode by `resolve_labels`. Therefore inserting a new `DEF(...)` opcode anywhere before the short-opcode block shifts `OP_nop` (and thus `OP_TEMP_START`), which shifts every short opcode's runtime number by 1. That changes the bytes of serialized bytecode for any function that uses a short opcode, so every hardcoded-bytecode test snapshot churns.

Placement matters for the size of the churn, not its existence: inserting just before `nop` (after `push_bigint_i32`) keeps all the normal mid-range opcodes at their numbers (only `nop` + never-serialized temps + the short opcodes shift), whereas inserting early (e.g. after `fclosure`) additionally renumbers every normal opcode in between. `nop` itself never reaches serialized bytecode (resolve_labels has `case OP_nop: /* remove erased code */ break;`), so it is safe to shift. The new opcode must stay before `nop` (line comment: "must be the last non short and non temporary opcode").

## BC_BASE_VERSION bump invalidates ALL serialized bytecode; the reader hard-rejects

`#define BC_BASE_VERSION` lives just below the `BCTagEnum`. The reader compares the version byte and throws `SyntaxError: invalid version (N expected=M)` on mismatch. Bumping it means every piece of bytecode the runtime loads must be regenerated by the same build.

Build-embedded bytecode (repl, inspect, module-impl, etc.) is regenerated at build time (the `qjsc`/`qjsc-minimal` ninja rules write to `build/intermediate/*.c`), so a plain rebuild handles those. The risk is CHECKED-IN bytecode. A grep over `*.c`/`*.h` is NOT sufficient: `tests/qjsbootstrap-bytecode.test.ts` carries a hardcoded `Buffer.from([...])` bytecode blob (regenerate via `require("quickjs:bytecode").fromFile("./tests/fixtures/log-four.js")`). When bumping the version, regenerate that blob too. (Before this work it was already stale at v6 and the test snapshotted a buffer-overrun error rather than executing; it now executes and prints 4.)

## The parse path is length-bounded, not NUL-terminated (so payload NULs are fine)

`js_parse_init` sets `buf_end = input + input_len`; `next_token` / `js_parse_string` / the new `js_parse_array_buffer_literal` all gate on `p >= s->buf_end`, never treating an interior `\0` as end of source. The new literal helper consumes exactly N payload bytes by count, so NUL (and bytes equal to 0x02/0x03) in the payload are fine.

The one module-detection scanner, `simple_next_token` (used by `peek_token` and `JS_DetectModule`), was NUL-reliant (`while (*p && ...)` comment loops, `c = *p++` with no end check). It now takes a `buf_end` parameter, gates on it, and has a `case 0x01:` that skips a well-formed literal by length so an interior NUL in a leading literal cannot truncate detection. Only two callers exist (both updated).

## Input channels that can carry interior NUL

Source files (`.js`/`.mjs`, imported modules, read via `QJU_ReadFile` which tracks `buf_len` separately from the convenience trailing `\0`) and precompiled bytecode (`BC_TAG_ARRAY_BUFFER` is length-prefixed) preserve NUL end to end. `-e`/argv, the std bootstrap snippet, and the interactive REPL are C-string / `strlen`-bounded and cannot carry an interior `0x00`.

## ArrayBuffer cpool constant round-trips; OP_push_array_buffer copies per eval

An ArrayBuffer is a first-class serializable bytecode constant (`BC_TAG_ARRAY_BUFFER`; `JS_WriteArrayBuffer` / `JS_ReadArrayBuffer`, which copies). The literal builds one cpool ArrayBuffer constant at parse time (`JS_NewArrayBufferCopy`, which copies out of the transient source buffer) and emits `OP_push_array_buffer idx`, whose interpreter CASE does another `JS_NewArrayBufferCopy` from the constant so each evaluation yields an independent buffer (fresh-per-eval, like `[]`/`{}`). The source `payload` pointer is never retained past the helper.

## Test environment: wine/wasm under Conductor

`shouldRunWineTests`/`shouldRunWasmTests` in `tests/_utils.ts` are `!process.env.CI`, so `CI=1 npx vitest run` skips both. Useful when running from a context where launching `wine` (`/opt/homebrew/bin/wine` -> `Wine Stable.app`) triggers macOS Gatekeeper prompts.

Verified cause (not a wrong-binary/PATH issue; there is only one `wine` on PATH): `Wine Stable.app` is unsigned and quarantined. `spctl -a -t exec "/Applications/Wine Stable.app"` reports `rejected, source=no usable signature`, and `com.apple.quarantine` is present on both the app and the inner `wine` binary. Gatekeeper allows it from Terminal/iTerm/VSCode because those are registered as Developer Tools (the `kTCCServiceDeveloperTool` privilege lets an app spawn code that fails Gatekeeper assessment without prompting); Conductor is not, so each spawn prompts.

Fixes (user action, not done automatically): either remove the quarantine flag once with `xattr -dr com.apple.quarantine "/Applications/Wine Stable.app"` (simplest, works regardless of launcher), or grant Conductor the Developer Tools privilege under System Settings > Privacy & Security > Developer Tools.
