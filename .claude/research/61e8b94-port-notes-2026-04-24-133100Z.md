# Key finding: fork's BigInt is always-on, not CONFIG_BIGNUM-gated

The plan in `~/.claude/plans/encapsulated-jumping-knuth.md` describes Phase A as "delete every `#ifdef CONFIG_BIGNUM` block that doesn't touch JS_TAG_BIG_INT/JS_CLASS_BIG_INT." My classifier found 122 such blocks, of which 4 touched BIG_INT. I tried to delete the other 118 blocks (keeping the 4). **The resulting code failed to compile** with ~20 errors about undeclared functions (`js_atof`, `is_math_mode`, `js_call_binary_op_simple`, `JS_NewBigFloat`, `JS_GetBigFloat`, `ctx->fp_env`, etc.).

Reason: many support functions for BigInt live INSIDE `#ifdef CONFIG_BIGNUM` blocks but are CALLED from outside those blocks — they were kept conditional in the fork because historically the fork may have supported disabling BigInt via CONFIG_BIGNUM. But `JS_AddIntrinsicBigInt` at line 52450 and the whole BigInt infrastructure are NOT under any `#ifdef CONFIG_BIGNUM`.

## What actually works

With `-DCONFIG_BIGNUM` undefined and `libbf` still linked:
- BigInt works fully (including `(1n << 100n).toString()`)
- `typeof BigFloat === "undefined"` (no CLI flag to enable it)
- `typeof 1.5 === "number"`

So Phase A (as the plan intends it) is effectively achievable by **only changing the outer scaffolding**:
- Remove `-DCONFIG_BIGNUM` from `meta/ninja/defs.ninja.js`
- Remove `--bignum` / `--qjscalc` flags from `src/programs/qjs/qjs.c`
- Remove `JS_AddIntrinsicBigFloat/Decimal/Operators/EnableBignumExt` calls from qjs/qjsc/qjsbootstrap/quickjs-run
- Remove bigfloat/bigdecimal/operators/useMath options from `src/builtin-modules/quickjs-context/quickjs-context.c` + `.d.ts`
- Delete `src/programs/qjscalc/` + unlink `qjscalc.target.o` from `src/programs/qjs/qjs.ninja.js`
- Leave `src/quickjs/quickjs.c`, `quickjs.h`, `quickjs-atom.h`, `quickjs-opcode.h`, `src/lib/libbf/` UNTOUCHED

The `#ifdef CONFIG_BIGNUM` blocks in quickjs.c simply become unreachable dead code (but still present, compiling fine since the ifdef is false). That's a valid Phase A intermediate.

**Do not try to surgically delete `#ifdef CONFIG_BIGNUM` blocks in Phase A.** The fork's BigFloat/BigDecimal/Operators/math-mode code is too intertwined with utility functions; partial deletion breaks compilation.

## Phase B is the hard part

Phase B needs to replace the libbf-backed BigInt wholesale with upstream's new short+limb-array BigInt. This means:
- Delete `src/lib/libbf/` + unlink from archives
- Delete `#include "libbf.h"` + all bf_t/bfdec_t uses in quickjs.c (hundreds of sites)
- Delete the entire remaining dead `#ifdef CONFIG_BIGNUM` code (the ~1400-line BigFloat/BigDecimal/Operators block at lines 51814-53876 post-baseline-1f554e4)
- Delete the fork's BigInt implementation (JS_AddIntrinsicBigInt body, js_bigint_to_string, js_binary_arith_bigint, etc. — all inside CONFIG_BIGNUM blocks currently but reachable without the ifdef)
- Port upstream's new BigInt: JSBigInt struct, ~5000 lines of arithmetic helpers, new serializers, new push_bigint_i32 executor case, tag renumber, BC_BASE_VERSION bump, `__JS_NewShortBigInt` macros across 3 JSValue layouts
- Add no-op stubs for `JS_AddIntrinsicBigFloat/Decimal/Operators/EnableBignumExt` (per user decision #3)
- Adapt ~30-50 `JS_Throw*Error` sites in ported code to fork's `(ctx, file, line, fmt, ...)` signature

### Scaffolding that can be added standalone (safe additions)

These changes work on their own without breaking anything — they just add future-facing plumbing that the current libbf BigInt doesn't use:

- `src/lib/cutils/cutils.h`: add `float64_as_uint64` / `uint64_as_float64` helpers (verbatim from upstream)
- `src/quickjs/quickjs.h`: add `JS_TAG_SHORT_BIG_INT=7`, shift `JS_TAG_FLOAT64` to 8, add `JS_LIMB_BITS`, `JS_SHORT_BIG_INT_BITS`, `__JS_NewShortBigInt`, `JS_VALUE_GET_SHORT_BIG_INT` (3 branches), update `JS_IsBigInt`
- `src/quickjs/quickjs-atom.h`: delete `bigfloat`, `bigdecimal`, `roundingMode`, `maximumSignificantDigits`, `maximumFractionDigits`, `BigFloat`, `BigFloatEnv`, `BigDecimal`, `OperatorSet`, `Operators`, `Symbol_operatorSet` atoms
- `src/quickjs/quickjs-opcode.h`: add `push_bigint_i32` opcode (and matching case in executor at line ~16525 in quickjs.c)
- `src/quickjs/quickjs.c`: bump `BC_BASE_VERSION 3→4` (unified regardless of CONFIG_BIGNUM)
- `src/quickjs/quickjs.h`: delete `JS_CLASS_BIG_FLOAT/FLOAT_ENV/BIG_DECIMAL/OPERATOR_SET` from class enum, delete `JS_IsBigFloat`/`JS_IsBigDecimal` helpers

I verified these scaffolding changes compile cleanly and pass the Phase-A-level smoke tests while libbf-backed BigInt continues to work (since upstream's new BigInt never runs — the `push_bigint_i32` opcode is never emitted yet).

But **the bytecode-version bump alone breaks all 3 bytecode-fixture tests** (as the plan warns). So scaffolding alone doesn't buy us Phase A; it's a partial Phase B that still needs the full BigInt replacement before fixtures can be regenerated and tests pass.

### Key risk: the `OP_push_bigint_i32` executor handler

Must sit right after `OP_push_i32` in the executor's switch:

```c
CASE(OP_push_bigint_i32):
    *sp++ = __JS_NewShortBigInt(ctx, (int)get_u32(pc));
    pc += 4;
    BREAK;
```

Without this, `dispatch_table[256]` references `case_OP_push_bigint_i32:` which doesn't exist and the build fails.

### Key risk: OP_push_bigint_i32 use sites in compiler

Upstream also emits the opcode in the bytecode compiler and references it in peephole optimization passes. Those sites need porting too — search upstream-postmerge-quickjs.c for `OP_push_bigint_i32` to find emitter/optimizer sites.

## Plan adjustments recommended

Phase A should be re-scoped: **just the outer scaffolding** (CLI flags, options, qjscalc dir), leaving internal `#ifdef CONFIG_BIGNUM` blocks as dead-but-present code. The plan's Phase A step 1 description that says "delete every `#ifdef CONFIG_BIGNUM` block that doesn't touch `JS_TAG_BIG_INT` / `JS_CLASS_BIG_INT`" is misleading — **don't do that**, because it breaks compilation due to the BigInt support functions living inside those blocks.

Phase B is the big, atomic port: BigInt replacement + `#ifdef CONFIG_BIGNUM` block deletion + libbf removal, all together.
